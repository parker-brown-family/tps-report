▼ 1 — Problem Statement
💬 comment
🚨 Live exploit — any logged-in user can claim any supporter tier for free
handlePaymentReturn() in index.html reads ?payment=success&tier=founding directly from the URL and writes it to Supabase. No Stripe verification. Any user who is logged in can navigate to that URL manually and immediately receive a Founding supporter badge — no payment required. The function runs on every page load and checks only that the query param exists.
💬 comment
No feedback loop from Stripe after initial payment
When a subscriber cancels, their card fails, or they upgrade — Stripe fires webhook events, but there is no handler. Cancelled subscribers keep their badge indefinitely. Payment failures are silent. There is no way to revoke or update supporter status without a manual database edit.
💬 comment
Explicitly out of scope
Tier upgrades / downgrades mid-cycle (customer.subscription.updated), a Stripe customer portal, and full JWT-based auth on the checkout endpoint are all intentionally deferred. This PR closes the exploit and handles cancellation — the minimum viable safety net.
▼ 2 — Proposed Design
💬 comment
APPROACH
Add a new api/stripe-webhook.js Vercel function that verifies Stripe's HMAC signature, then writes supporter_tier (and stripe_customer_id) to Supabase using the service role key. Update create-checkout-session.js to pass client_reference_id (Supabase user ID), customer_email, and metadata: { tier } so the webhook can identify the right user without a second Stripe API call. Remove the Supabase write from handlePaymentReturn() — the webhook becomes the sole authoritative writer of supporter_tier. The optimistic UI (toast, badge display) is preserved so the experience still feels instant. 
// api/stripe-webhook.js — core event handler
case 'checkout.session.completed': {
  const { client_reference_id: userId,
          customer: stripeCustomerId,
          metadata } = event.data.object;
  if (userId && metadata?.tier) {
    await supabase.from('profiles')
      .update({
        supporter_tier: metadata.tier,
        stripe_customer_id: stripeCustomerId   // needed for subscription.deleted later
      })
      .eq('id', userId);
  }
  break;
}

case 'customer.subscription.deleted': {
  const stripeCustomerId = event.data.object.customer;
  await supabase.from('profiles')
    .update({ supporter_tier: null })
    .eq('stripe_customer_id', stripeCustomerId);
  break;
}
💬 comment
What we chose NOT to do, and why
No JWT verification on the checkout endpoint. The userId sent by the frontend is trusted for client_reference_id only. A contrived attack (user A pays, sends user B's ID) would credit user B — low risk in a small community app, deferred.   No subscription.updated handler. The event fires for trial endings, quantity changes, and many internal Stripe state changes — adding it without an upgrade UI would cause false positive badge changes.   Raw body via stream, no raw-body package. Vercel parses JSON by default; Stripe signature verification requires the raw bytes. We read the stream directly (3 lines) rather than adding a new dependency.
▼ 3 — Considerations
💬 comment
⚠️ Files touched by this change
index.html — handlePaymentReturn() loses its DB write; completeSupport() gains userId + email in the fetch body. These two functions are large and share surrounding auth state — take care with brace balance (the CLAUDE.md in that repo warns explicitly about silent crashes from mismatched braces).   api/create-checkout-session.js — adds 3 fields to the session; backward-compatible.   api/stripe-webhook.js — new file; must have handler.config = { api: { bodyParser: false } } or Stripe signature verification will always fail.
💬 comment
⚠️ Prerequisite: Supabase schema migration
The profiles table needs a new column before any code ships. Without it, the webhook's stripe_customer_id write silently fails and subscription.deleted can never find the right user to revoke. Run in Supabase SQL editor before deploying:

ALTER TABLE profiles ADD COLUMN stripe_customer_id TEXT;
💬 comment
Testing strategy
Exploit closure: after deploy, try navigating to /?payment=success&tier=founding while logged in — badge should appear briefly (optimistic UI), then disappear on refresh. Webhook: use Stripe CLI (stripe listen --forward-to localhost:3000/api/stripe-webhook) to replay checkout.session.completed and customer.subscription.deleted events locally. Verify supporter_tier is written and cleared in Supabase. Signature rejection: send a POST to the webhook endpoint without a valid stripe-signature header — must return 400.

▼ Open Questions
💬 comment
Q1
Should the optimistic badge display in handlePaymentReturn() be removed entirely, or is showing it before the webhook confirms acceptable?
If the webhook silently fails (misconfigured secret, Supabase outage), the user sees their supporter badge immediately — then loses it on next refresh with no explanation. Keeping optimistic UI is friendlier but masks webhook failures. Removing it makes the experience feel slower but honest. Keddy decides based on how much she trusts the webhook infrastructure.
💬 comment
Q2
What happens to users who became supporters before this PR ships — their stripe_customer_id will be NULL, so a future cancellation webhook won't find them. Is a one-time backfill needed?
Keddy likely knows who the current supporters are (small community, early days). Options: (a) ignore — when they cancel, their badge just stays until manually cleared; (b) backfill stripe_customer_id manually by cross-referencing Stripe's customer list with the profiles table. Option (a) is fine if there are only a handful of supporters right now.
